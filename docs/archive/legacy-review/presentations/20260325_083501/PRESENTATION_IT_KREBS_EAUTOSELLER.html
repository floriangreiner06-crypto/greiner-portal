Arbeitsweise

Cursor, Workstreams, Git, Regeln

Entwicklungsumgebung

  • Server ist Master: Entwicklung auf /opt/greiner-portal/
  • Windows-Sync (\\Srvrdb01\...\Server\) nur Backup; bei Bedarf vom Server ins Sync kopieren
  • Git lebt auf dem Server — keine force-pushes/rebase ohne Absprache
  • Commit-Konvention: feat(workstream): … / fix(workstream): … / docs(workstream): …

Workstream-System

Jedes Thema hat einen Workstream mit docs/workstreams/{workstream}/CONTEXT.md. Beim Arbeiten:

  1. CONTEXT.md des Workstreams lesen
  2. Im Kontext arbeiten
  3. CONTEXT.md am Ende aktualisieren

Workstreams: controlling, werkstatt, verkauf, teile-lager, urlaubsplaner, hr, planung, fahrzeugfinanzierungen, infrastruktur, auth-ldap, integrations (WhatsApp, eAutoSeller, SOAP, Scraper), marketing, verguetung, Hilfe

Cursor & Regeln

  • .cursorrules + CLAUDE.md — Architektur, Pfade, SSOT, Deployment, Navi-Regeln
  • Session-Start/Session-Ende über .cursor/commands/
  • Agent führt Migrationen und Neustarts selbst aus

Reporting: DRIVE + optional Metabase

DRIVE ist unsere eigenentwickelte Reporting-Plattform (TEK, BWA, Bankenspiegel, E-Mail-Reports, PDFs). Metabase (:3001) ergänzend für Ad-hoc-Abfragen und Überblick; Haupt-Reports laufen über DRIVE. MCP, Doku: docs/, docs/workstreams/.

Deployment (kurz)

# Nach Python-Änderungen sudo systemctl restart greiner-portal # Nach Celery-Task-Änderungen sudo systemctl restart celery-worker celery-beat # Migration ausführen PGPASSWORD=... psql -h 127.0.0.1 -U drive_user -d drive_portal -f migrations/<datei>.sql

Templates: nur Browser-Refresh (Strg+F5), kein Restart.
